## 3. 安全测试专项 (Security Testing)

> **本章为集成测试的安全维度的补充测试用例**，与功能联调并行执行。

### 3.1 输入验证安全测试

| # | 测试场景 | 测试步骤 | 预期结果 | 严重度 |
|---|----------|----------|----------|--------|
| SEC-01 | 症状字段 XSS 注入 | 在症状描述中输入 `<script>alert('xss')</script>` 或 `<img src=x onerror=alert(1)>` 并提交预约 | 预约成功创建，但在任何页面展示该预约的症状时**不会**触发脚本执行（文本插值生效） | 高 |
| SEC-02 | 症状字段超长输入 | 在症状描述中粘贴 2000 字符的超长文本 | 输入应被截断至 500 字符以内（前端限制或 Store 层截断），不导致页面崩溃或内存溢出 | 中 |
| SEC-03 | 特殊字符输入 | 症状中输入 SQL 注入风格文本（`'; DROP TABLE; --`）、HTML 实体（`&lt;&gt;`）、Emoji | 文本原样存储并在展示时作为纯文本渲染，不改变语义 | 低 |
| SEC-04 | 日期篡改尝试 | 使用浏览器 DevTools 修改 DatePicker 的值为过去日期或非法格式 | 表单校验应拦截或 Store 层 `createAppointment` 拒绝创建 | 中 |

### 3.2 身份验证与授权测试

| # | 测试场景 | 测试步骤 | 预期结果 | 严重度 |
|---|----------|----------|----------|--------|
| SEC-05 | 未验证用户直接访问 /appointment | 清除所有状态后直接在地址栏输入 `/appointment` | 页面仅显示身份验证表单，**不显示**任何医生列表、时段选择等预约功能界面 | 高 |
| SEC-06 | 患者 A 查看 患者 B 的预约 | 以患者 A 身份登录 → Consultation.vue → 我的预约 Tab | 仅列出患者 A 的预约，**无法**通过任何手段（Console 修改、URL 参数）查看到患者 B 的数据 | 高 |
| SEC-07 | 医生 A 操作医生 B 的预约 | 以医生 A 登录诊室 → 预约管理区域 | 仅显示分配给医生 A 的预约；**无法**确认/拒绝/完成医生 B 的预约 | 高 |
| SEC-08 | 未登录用户访问 DoctorRoom | 直接访问 `/doctor/room`（无 doctor 登录态） | 被 onMounted 守卫重定向至 `/doctor/login`，**无法**看到任何预约管理区域 | 高 |

### 3.3 业务逻辑安全测试

| # | 测试场景 | 测试步骤 | 预期结果 | 严重度 |
|---|----------|----------|----------|--------|
| SEC-09 | 重复提交预约（防重放） | 快速连续点击「提交预约」按钮 5 次（在首次响应返回前） | 仅创建 1 条预约记录；按钮在 loading 期间处于 disabled 态 | 中 |
| SEC-10 | 超过容量的时段预约 | 选择一个已有 3 条 non-cancelled 预约的时段，尝试再创建第 4 条 | Store 层 `createAppointment` 返回 null 或抛错；前端提示「该时段已约满」 | 高 |
| SEC-11 | <24h 取消预约规避 | 创建一个距当前时间不足 24 小时的预约 → 尝试取消 | 取消操作被拒绝，提示「距就诊不足 24 小时，无法取消」；**无法**通过 DevTools 修改状态绕过 | 中 |
| SEC-12 | 非法状态跃迁 | 在 DevTools Console 中直接调用 `store.confirmAppointment()` 对一个已经是 `completed` 状态的预约执行 | 操作被忽略或抛出明确错误；状态不变更为非法值 | 中 |
| SEC-13 | 不存在的 ID 操作 | 对一个不存在的 appointmentId 调用 `store.cancelAppointment('non-existent-id')` | 返回静默失败或 undefined；**不崩溃**、**不暴露**内部错误信息 | 低 |

### 3.4 信息泄露测试

| # | 测试场景 | 测试步骤 | 预期结果 | 严重度 |
|---|----------|----------|----------|--------|
| SEC-14 | Browser Console 信息泄露 | 打开 DevTools Console → 查看是否有 console.log 输出了完整预约对象、token、密码等敏感信息 | 不应有任何敏感数据的 console 输出（调试用 log 应在生产构建中被移除或使用环境变量控制） | 中 |
| SEC-15 | 网络请求敏感信息 | 打开 DevTools Network 面板 → 查看 XHR/Fetch 请求 | MVP 阶段无真实后端 API 请求；若有静态资源加载，URL 中不应包含 query params 携带 token/用户信息 | 低 |
| SEC-16 | 错误消息信息泄露 | 触发各种异常场景（网络断开模拟、无效输入等） | 所有面向用户的错误消息均为友好的中文提示，**不包含**堆栈跟踪、文件路径、SQL 语句、内部 API 结构等信息 | 中 |
| SEC-17 | URL 中的敏感参数 | 完成预约流程后观察浏览器地址栏 URL | URL 中不应出现 appointmentId、patientId、症状内容等敏感查询参数 | 低 |
